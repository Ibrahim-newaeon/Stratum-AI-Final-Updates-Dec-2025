3. Signal Health Calculations

Signal Health is a composite score (0-100) that evaluates data reliability before automation decisions. The score is calculated using weighted components that measure different aspects of data quality.

Composite Score Formula

Signal Health = (EMQ × 0.40) + (Freshness × 0.25) + (Variance × 0.20) + (Anomaly × 0.15) + [CDP_EMQ × 0.10]

Where:
├── EMQ Score: Event Match Quality from advertising platforms (0-100)
├── Freshness Score: Data recency measurement (0-100)
├── Variance Score: Attribution alignment between platforms (0-100)
├── Anomaly Score: Z-score based anomaly detection (0-100)
└── CDP EMQ: Optional Customer Data Platform quality score (0-100)

EMQ (Event Match Quality) - 5-Driver Model

EMQ evaluates the quality of event data from advertising platforms using five weighted components:

1. Event Match Rate (30%)

Measures overlap between Pixel and CAPI (Conversions API) events.

  • ≥85% = GOOD
  • 70-85% = WARNING
  • <70% = CRITICAL

2. Pixel Coverage (25%)

Evaluates tracking pixel installation completeness.

  • ≥90% = GOOD
  • 75-90% = WARNING
  • <75% = CRITICAL

3. Conversion Latency (20%)

Measures time delay in conversion reporting.

<1h = 100 | <4h = 70-90
<12h = 40-70 | <24h = 0-40
>24h = 0

4. Attribution Accuracy (15%)

Platform vs GA4 conversion alignment.

  • Variance <10% = 80+
  • 10-50% = 60-80
  • >50% = <60

5. Data Freshness (10%)

Time since last data update received.

<1h = 100 | <6h = 80-96
<24h = 40-80 | >24h = 0-40

5. Anomaly Detection

The system uses Z-Score analysis to detect unusual patterns in marketing metrics, automatically identifying outliers that may indicate problems or opportunities.

Z-Score Algorithm

Z-Score = (X - μ) / σ

Where:
├── X = Current metric value
├── μ = Mean of baseline period (last 14 days)
└── σ = Standard deviation of baseline period

Baseline Configuration:
└── Window: 14 days of historical data

Severity Thresholds

6. RFM Customer Segmentation

RFM (Recency, Frequency, Monetary) analysis uses quintile-based scoring to segment customers into actionable groups for targeted marketing.

RFM Scoring Method

Each customer receives a 1-5 score for each dimension:

Recency (R): Days since last order (inverted - lower days = higher score)
├── Quintile 5: Most recent purchasers
├── Quintile 1: Longest since purchase

Frequency (F): Total number of orders
├── Quintile 5: Most frequent buyers
├── Quintile 1: One-time purchasers

Monetary (M): Total revenue generated
├── Quintile 5: Highest spenders
└── Quintile 1: Lowest spenders

8. Confidence Scoring

Every prediction includes a confidence score to quantify uncertainty and guide decision-making.

Confidence Calculation Components

Base Confidence = 0.50

Additive Factors:
├── Sample Size: +0.30 for 1 campaign, +0.60 for 10+ campaigns
├── Data Completeness: +0.20 for complete spend/revenue/ROAS data
├── Order History: +0.25 if customer has 3+ orders
├── Engagement: +0.10 per engagement type (email, session, etc.)

Subtractive Factors:
├── High Variance: -0.20 for high ROAS variance
├── Data Gaps: -0.15 for missing key metrics

Final Confidence = min(Calculated Score, 0.95) // Never claim 100%

11. Red Flags Checklist for Non-Technical Reviewers

This checklist helps non-technical stakeholders identify potential problems in a software project. Use this to evaluate project health during reviews, due diligence, or ongoing oversight.

Project Health Indicators

Green Flags (Positive Signs)

  • Regular code updates (commits at least weekly)
  • Tests exist and pass consistently
  • Documentation is present and updated
  • Multiple developers have contributed
  • Issues and bugs are tracked and resolved
  • Version control (Git) is used properly
  • Security updates are applied promptly
  • Environment configurations are separate from code

Red Flags (Warning Signs)

  • No recent updates (months of inactivity)
  • No tests or tests frequently failing
  • Missing or outdated documentation
  • Single developer with no backup
  • Known bugs ignored for long periods
  • No version control or messy history
  • Outdated dependencies with known vulnerabilities
  • Passwords/secrets visible in code
